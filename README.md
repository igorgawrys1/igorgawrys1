## What I Like Building

I like systems that remove operational drag:

- internal platforms that make teams faster,
- AI assistants that can use tools instead of only generating text,
- support automation with confidence checks and escalation paths,
- developer workflows where agents can code, test, and ship safely,
- business software that is boring in the best way: reliable, understandable, and maintainable.

My default approach is pragmatic. Start with the workflow, understand the failure modes, build the smallest useful system, then harden it with tests, observability, permissions, and fallbacks.

## How I Think About AI

AI is most useful when it is connected to context, tools, rules, and feedback loops.

Enterprise AI governance matters here: privacy rules, data boundaries, auditability, fallback paths, and human control are part of the product, not paperwork after the fact.

I care less about prompting tricks and more about systems design: what data can the model see, what tools can it call, what should happen when confidence is low, what gets logged, what must stay private, and how humans stay in control.

That is also how I use AI in engineering: as a force multiplier around code search, implementation, testing, documentation, operations, and repeated workflows - with the assumption that generated output still needs senior-level review.
